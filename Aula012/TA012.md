## 1. Conceitos Fundamentais

### O Problema do Deployment Manual

Imagine que você desenvolveu uma aplicação e precisa colocá-la em produção. O processo manual seria:
1. Fazer build da aplicação
2. Executar testes
3. Criar imagem Docker
4. Fazer push para registry
5. Fazer deploy no servidor
6. Verificar se tudo funcionou

Agora imagine fazer isso 10 vezes por dia, para múltiplos ambientes, com múltiplos desenvolvedores. O processo manual se torna um gargalo, propenso a erros e inconsistente.

**CI/CD** resolve esse problema automatizando todo o pipeline de entrega de software.

### O que é CI/CD?

#### Continuous Integration (CI)
**Integração Contínua** é a prática de integrar mudanças de código frequentemente:
- **Commits frequentes**: Múltiplas vezes por dia
- **Build automático**: Cada commit dispara build
- **Testes automáticos**: Validação automática do código
- **Feedback rápido**: Desenvolvedores sabem imediatamente se algo quebrou

#### Continuous Delivery (CD)
**Entrega Contínua** estende CI para automatizar o deployment:
- **Deploy automático**: Para ambientes de teste/staging
- **Deploy manual**: Para produção (aprovação humana)
- **Rollback rápido**: Capacidade de reverter mudanças
- **Ambientes consistentes**: Mesmo processo para todos os ambientes

#### Continuous Deployment
**Deployment Contínuo** vai além, automatizando também produção:
- **Deploy totalmente automático**: Até produção
- **Sem intervenção humana**: Baseado em critérios automáticos
- **Confiança alta**: Requer testes muito robustos
- **Feedback do usuário**: Monitoramento em tempo real

### Benefícios do CI/CD

#### Para Desenvolvedores
- **Feedback rápido**: Problemas detectados cedo
- **Menos conflitos**: Integrações frequentes
- **Foco no código**: Menos tempo com deployment
- **Confiança**: Processo consistente e testado

#### Para o Negócio
- **Time to Market**: Features chegam mais rápido ao usuário
- **Qualidade**: Menos bugs em produção
- **Custos**: Menos tempo gasto com problemas
- **Competitividade**: Resposta rápida ao mercado

### Amazon ECR (Elastic Container Registry)

#### O que é ECR?

**Amazon ECR** é um registry Docker totalmente gerenciado que facilita armazenar, gerenciar e deployar imagens Docker:

**Características:**
- **Totalmente gerenciado**: Sem infraestrutura para manter
- **Seguro**: Integração com IAM, encryption at rest/transit
- **Escalável**: Suporta qualquer volume de imagens
- **Integrado**: Funciona nativamente com ECS, EKS, Lambda
- **Confiável**: Alta disponibilidade e durabilidade

#### ECR vs Docker Hub

| Aspecto | Docker Hub | Amazon ECR |
|---------|------------|------------|
| **Custo** | Gratuito (limitado) | Pago por uso |
| **Segurança** | Básica | Avançada (IAM) |
| **Performance** | Variável | Consistente |
| **Integração AWS** | Limitada | Nativa |
| **Compliance** | Básico | Enterprise |

#### Comandos ECR Essenciais

```bash
# Autenticar Docker com ECR
aws ecr get-login-password --region us-east-1 | \
  docker login --username AWS --password-stdin \
  123456789012.dkr.ecr.us-east-1.amazonaws.com

# Criar repositório
aws ecr create-repository --repository-name minha-app

# Listar repositórios
aws ecr describe-repositories

# Tag e push da imagem
docker tag minha-app:latest \
  123456789012.dkr.ecr.us-east-1.amazonaws.com/minha-app:latest

docker push 123456789012.dkr.ecr.us-east-1.amazonaws.com/minha-app:latest

# Listar imagens no repositório
aws ecr list-images --repository-name minha-app
```

### Pipeline CI/CD com AWS

#### Arquitetura Típica

```
Developer → Git Push → CodeCommit → CodeBuild → ECR → CodeDeploy → ECS/EKS
    ↓           ↓           ↓           ↓        ↓        ↓
  Local     Source      Build &     Store    Deploy   Production
  Dev       Control     Test       Images   Artifacts  Environment
```

#### AWS CodeCommit
**CodeCommit** é o serviço de controle de versão Git da AWS:

```bash
# Configurar credenciais
aws configure set region us-east-1

# Clonar repositório
git clone https://git-codecommit.us-east-1.amazonaws.com/v1/repos/minha-app

# Operações Git normais
git add .
git commit -m "Nova feature"
git push origin main
```

#### AWS CodeBuild
**CodeBuild** é o serviço de build gerenciado:

**buildspec.yml**:
```yaml
version: 0.2

phases:
  pre_build:
    commands:
      - echo Logging in to Amazon ECR...
      - aws ecr get-login-password --region $AWS_DEFAULT_REGION | docker login --username AWS --password-stdin $AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com
      - REPOSITORY_URI=$AWS_ACCOUNT_ID.dkr.ecr.$AWS_DEFAULT_REGION.amazonaws.com/$IMAGE_REPO_NAME
      - COMMIT_HASH=$(echo $CODEBUILD_RESOLVED_SOURCE_VERSION | cut -c 1-7)
      - IMAGE_TAG=${COMMIT_HASH:=latest}
  
  build:
    commands:
      - echo Build started on `date`
      - echo Building the Docker image...
      - docker build -t $REPOSITORY_URI:latest .
      - docker tag $REPOSITORY_URI:latest $REPOSITORY_URI:$IMAGE_TAG
  
  post_build:
    commands:
      - echo Build completed on `date`
      - echo Pushing the Docker images...
      - docker push $REPOSITORY_URI:latest
      - docker push $REPOSITORY_URI:$IMAGE_TAG
      - echo Writing image definitions file...
      - printf '[{"name":"minha-app","imageUri":"%s"}]' $REPOSITORY_URI:$IMAGE_TAG > imagedefinitions.json

artifacts:
  files:
    - imagedefinitions.json
```

#### AWS CodePipeline
**CodePipeline** orquestra todo o pipeline:

```yaml
# pipeline.yml (CloudFormation)
Resources:
  Pipeline:
    Type: AWS::CodePipeline::Pipeline
    Properties:
      RoleArn: !GetAtt CodePipelineRole.Arn
      Stages:
        - Name: Source
          Actions:
            - Name: SourceAction
              ActionTypeId:
                Category: Source
                Owner: AWS
                Provider: CodeCommit
                Version: 1
              Configuration:
                RepositoryName: !Ref RepoName
                BranchName: main
              OutputArtifacts:
                - Name: SourceOutput
        
        - Name: Build
          Actions:
            - Name: BuildAction
              ActionTypeId:
                Category: Build
                Owner: AWS
                Provider: CodeBuild
                Version: 1
              Configuration:
                ProjectName: !Ref CodeBuildProject
              InputArtifacts:
                - Name: SourceOutput
              OutputArtifacts:
                - Name: BuildOutput
        
        - Name: Deploy
          Actions:
            - Name: DeployAction
              ActionTypeId:
                Category: Deploy
                Owner: AWS
                Provider: ECS
                Version: 1
              Configuration:
                ClusterName: !Ref ECSCluster
                ServiceName: !Ref ECSService
              InputArtifacts:
                - Name: BuildOutput
```

### GitHub Actions com ECR

#### Workflow Example
```yaml
# .github/workflows/deploy.yml
name: Deploy to ECR

on:
  push:
    branches: [ main ]

env:
  AWS_REGION: us-east-1
  ECR_REPOSITORY: minha-app

jobs:
  deploy:
    name: Deploy
    runs-on: ubuntu-latest
    
    steps:
    - name: Checkout
      uses: actions/checkout@v3

    - name: Configure AWS credentials
      uses: aws-actions/configure-aws-credentials@v2
      with:
        aws-access-key-id: ${{ secrets.AWS_ACCESS_KEY_ID }}
        aws-secret-access-key: ${{ secrets.AWS_SECRET_ACCESS_KEY }}
        aws-region: ${{ env.AWS_REGION }}

    - name: Login to Amazon ECR
      id: login-ecr
      uses: aws-actions/amazon-ecr-login@v1

    - name: Build, tag, and push image to Amazon ECR
      id: build-image
      env:
        ECR_REGISTRY: ${{ steps.login-ecr.outputs.registry }}
        IMAGE_TAG: ${{ github.sha }}
      run: |
        docker build -t $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG .
        docker push $ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG
        echo "image=$ECR_REGISTRY/$ECR_REPOSITORY:$IMAGE_TAG" >> $GITHUB_OUTPUT

    - name: Deploy to ECS
      uses: aws-actions/amazon-ecs-deploy-task-definition@v1
      with:
        task-definition: task-definition.json
        service: minha-app-service
        cluster: minha-app-cluster
        wait-for-service-stability: true
```

### Estratégias de Deployment

#### Blue-Green Deployment
```yaml
# Duas versões completas do ambiente
Blue Environment (Produção Atual)
Green Environment (Nova Versão)

# Switch instantâneo via load balancer
Load Balancer → Blue (100% traffic)
Load Balancer → Green (0% traffic)

# Após validação
Load Balancer → Green (100% traffic)
Load Balancer → Blue (0% traffic)
```

#### Canary Deployment
```yaml
# Deployment gradual
Versão Atual: 90% do tráfego
Versão Nova: 10% do tráfego

# Aumentar gradualmente
Versão Atual: 70% do tráfego
Versão Nova: 30% do tráfego

# Até completar
Versão Atual: 0% do tráfego
Versão Nova: 100% do tráfego
```

#### Rolling Deployment
```yaml
# Substituição gradual de instâncias
Instâncias: [V1, V1, V1, V1]
Instâncias: [V2, V1, V1, V1]
Instâncias: [V2, V2, V1, V1]
Instâncias: [V2, V2, V2, V1]
Instâncias: [V2, V2, V2, V2]
```

## 2. Questões de Múltipla Escolha

### Questão 1
Qual é a principal diferença entre Continuous Delivery e Continuous Deployment?

a) Continuous Delivery é mais rápido que Continuous Deployment
b) Continuous Delivery requer aprovação manual para produção, Continuous Deployment é totalmente automático
c) Continuous Delivery é para desenvolvimento, Continuous Deployment é para produção
d) Não há diferença, são sinônimos

**Gabarito: B**

**Justificativa:** Continuous Delivery automatiza o pipeline até o ponto onde o código está pronto para produção, mas requer aprovação manual para deploy em produção. Continuous Deployment vai além, automatizando também o deploy para produção sem intervenção humana, baseado em critérios automáticos como testes passando e métricas de qualidade.

### Questão 2
Por que usar Amazon ECR em vez de Docker Hub para aplicações AWS?

a) ECR é gratuito enquanto Docker Hub é pago
b) ECR oferece melhor integração com serviços AWS e segurança avançada
c) ECR é mais rápido que Docker Hub
d) Docker Hub não suporta imagens Docker

**Gabarito: B**

**Justificativa:** Amazon ECR oferece integração nativa com serviços AWS (ECS, EKS, Lambda), segurança avançada via IAM, compliance enterprise, e performance consistente dentro da AWS. Embora Docker Hub seja gratuito para uso básico, ECR fornece recursos empresariais essenciais para workloads de produção na AWS, como encryption, vulnerability scanning e controle de acesso granular.

### Questão 3
Em uma estratégia Blue-Green deployment, qual é a principal vantagem?

a) Usa menos recursos que outras estratégias
b) Permite rollback instantâneo e zero downtime
c) É mais simples de implementar
d) Reduz o tempo de build

**Gabarito: B**

**Justificativa:** Blue-Green deployment mantém duas versões completas do ambiente (blue = atual, green = nova). O switch é instantâneo via load balancer, proporcionando zero downtime. Se houver problemas, o rollback é imediato (switch de volta). Embora use mais recursos (duas versões completas), oferece máxima confiabilidade e capacidade de rollback rápido.
