## Passo 2: O Arquivo de Configuração do Nginx

### Criando o nginx.conf Personalizado

```bash
code .
```

**No Arquivo nginx.conf Digite o seguinte conteúdo:**

```nginx
events {
    worker_connections 1024;
}

http {
    # Definir o grupo de servidores backend
    upstream backend_cluster {
        # Algoritmo padrão: Round Robin
        server app1:80;
        server app2:80;
        
        # Opções avançadas (comentadas para demonstração)
        # least_conn;                    # Algoritmo: Least Connections
        # server app1:80 weight=3;       # Peso maior para app1
        # server app2:80 backup;         # app2 como backup
    }

    # Configuração de logs personalizada
    log_format upstream_log '$remote_addr - $remote_user [$time_local] '
                           '"$request" $status $body_bytes_sent '
                           '"$http_referer" "$http_user_agent" '
                           'upstream_addr: $upstream_addr '
                           'upstream_response_time: $upstream_response_time';

    server {
        listen 80;
        server_name localhost;

        # Log de acesso com formato personalizado
        access_log /var/log/nginx/access.log upstream_log;

        location / {
            # Proxy para o cluster backend
            proxy_pass http://backend_cluster;
            
            # Headers importantes para proxy reverso
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto $scheme;
            
            # Configurações de timeout
            proxy_connect_timeout 30s;
            proxy_send_timeout 30s;
            proxy_read_timeout 30s;
        }

        # Endpoint de health check
        location /health {
            access_log off;
            return 200 "Load Balancer OK\n";
            add_header Content-Type text/plain;
        }
    }
}
```

### Entendendo a Configuração

**upstream backend_cluster**
- Define um grupo de servidores backend
- Nginx distribui requisições entre eles automaticamente

**Algoritmos de Balanceamento:**
- **Round Robin** (padrão): Distribui requisições sequencialmente
- **least_conn**: Direciona para servidor com menos conexões ativas
- **ip_hash**: Mantém sessão baseada no IP do cliente

**proxy_pass http://backend_cluster**
- Encaminha requisições para o grupo upstream
- Nginx escolhe automaticamente qual servidor usar

**Headers de Proxy:**
- `X-Real-IP`: IP original do cliente
- `X-Forwarded-For`: Cadeia de proxies
- `X-Forwarded-Proto`: Protocolo original (HTTP/HTTPS)

### ⚠️ Atenção: Sintaxe do Nginx

```nginx
# ✅ CORRETO - Ponto e vírgula obrigatório
server app1:80;
proxy_pass http://backend_cluster;

# ❌ ERRADO - Sem ponto e vírgula
server app1:80
proxy_pass http://backend_cluster
```

> [!IMPORTANT]
>
> **Importante:** Nginx exige ponto e vírgula (;) no final de cada diretiva. Esquecer causa erro de sintaxe!

## Passo 3: Docker Compose com Múltiplas Instâncias

### Criando o docker-compose.yml

**No arquivo docker-compose.ymlDigite o seguinte conteúdo:**

```yaml
version: '3.8'

services:
  # Load Balancer (Nginx Proxy)
  proxy:
    image: nginx:alpine
    container_name: nginx-proxy
    ports:
      - "8080:80"
    volumes:
      - ./nginx.conf:/etc/nginx/nginx.conf:ro
    networks:
      - app-network
    depends_on:
      - app1
      - app2
    restart: always

  # Primeira instância da aplicação
  app1:
    image: nginxdemos/hello
    container_name: app1-instance
    networks:
      - app-network
    restart: always
    environment:
      - INSTANCE_NAME=APP-1

  # Segunda instância da aplicação
  app2:
    image: nginxdemos/hello
    container_name: app2-instance
    networks:
      - app-network
    restart: always
    environment:
      - INSTANCE_NAME=APP-2

networks:
  app-network:
    driver: bridge
```

### Explicação dos Serviços

**nginxdemos/hello**
- Imagem especial que mostra informações do container
- Exibe hostname, IP e outras informações úteis
- Perfeita para visualizar o balanceamento de carga

**volumes: ./nginx.conf:/etc/nginx/nginx.conf:ro**
- Monta nossa configuração personalizada
- `:ro` = read-only (boa prática de segurança)

**depends_on: app1, app2**
- Garante que aplicações iniciem antes do proxy
- Evita erros de "upstream not found"

## Algoritmos de Balanceamento

### Round Robin (Padrão)

```nginx
upstream backend_cluster {
    server app1:80;
    server app2:80;
}
```
- Distribui requisições sequencialmente
- Simples e eficiente para cargas similares

### Least Connections

```nginx
upstream backend_cluster {
    least_conn;
    server app1:80;
    server app2:80;
}
```
- Direciona para servidor com menos conexões ativas
- Melhor para requisições de duração variável

### Weighted Round Robin

```nginx
upstream backend_cluster {
    server app1:80 weight=3;
    server app2:80 weight=1;
}
```
- app1 recebe 75% das requisições
- app2 recebe 25% das requisições

### IP Hash

```nginx
upstream backend_cluster {
    ip_hash;
    server app1:80;
    server app2:80;
}
```
- Mantém sessão baseada no IP do cliente
- Útil para aplicações com estado (sessões)

## Conceitos Avançados (Bônus)

### Health Checks no Upstream

```nginx
upstream backend_cluster {
    server app1:80 max_fails=3 fail_timeout=30s;
    server app2:80 max_fails=3 fail_timeout=30s;
}
```

### Sticky Sessions

```nginx
upstream backend_cluster {
    ip_hash;
    server app1:80;
    server app2:80;
}
```

### Backup Servers

```nginx
upstream backend_cluster {
    server app1:80;
    server app2:80 backup;
}
```



## Conexão com AWS

**Conceitos aplicáveis na AWS:**

- **Application Load Balancer (ALB)**: Equivalente ao nosso Nginx proxy
- **Target Groups**: Equivalente ao nosso upstream
- **Health Checks**: Monitoramento automático de instâncias
- **Auto Scaling**: Adiciona/remove instâncias automaticamente
- **ECS Service**: Gerencia múltiplas instâncias de containers
