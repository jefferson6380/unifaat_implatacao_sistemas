## Passo 1: Protegendo a "Joia da Coroa" (Conta Root)

### O que é a Conta Root?

A conta root da AWS é como o usuário `root` no Linux - tem acesso **TOTAL** a todos os recursos e configurações. Por isso, deve ser protegida como um cofre.

### Ativando MFA (Multi-Factor Authentication)

**1. Acesse o Console AWS:**
- Faça login em: https://aws.amazon.com/console/
- Use suas credenciais de conta root (e-mail + senha)

**2. Navegue para IAM:**
- No console, procure por "IAM" na barra de pesquisa
- Clique em "IAM" (Identity and Access Management)

**3. Configure MFA para Root:**
- No painel IAM, procure por "Security recommendations"
- Clique em "Add MFA for root user"
- Ou vá em: Account settings → Security credentials → Multi-factor authentication (MFA)

**4. Escolha o Tipo de MFA:**
- **Recomendado:** Authenticator app (Google Authenticator, Authy, Microsoft Authenticator)
- **Alternativa:** SMS (menos seguro, mas funcional)

**5. Configure o Authenticator:**
- Escaneie o QR Code com seu app
- Digite dois códigos consecutivos para validar
- **Salve os códigos de backup** em local seguro

### REGRA DE OURO: Access Keys na Conta Root

**❌ NUNCA FAÇA ISSO:**
```bash
# JAMAIS crie Access Keys para a conta root
# Isso é uma falha crítica de segurança
```

**✅ POR QUE NÃO:**
- **Risco máximo**: Access Keys da root têm poder total
- **Sem rotação**: Difícil de gerenciar e rotacionar
- **Auditoria**: Impossível rastrear quem usou
- **Vazamento**: Se vazar no GitHub, sua conta está comprometida

**✅ O CORRETO:**
- Conta root apenas para tarefas administrativas críticas
- Usar usuários IAM para operações diárias
- MFA sempre ativo na conta root



## Passo 2: Gestão de Custos (Billing)

### Por que Monitorar Custos?

Na nuvem, recursos são cobrados por uso. Um erro de configuração pode gerar custos inesperados. **Monitoramento de custos é parte da engenharia.**

### Criando Billing Alarm

**1. Acesse CloudWatch:**
- Console AWS → CloudWatch
- **Importante:** Mude para região **US East (N. Virginia)** - us-east-1
- Billing metrics só funcionam nesta região

**2. Habilite Billing Alerts:**
- AWS Console → Account → Billing preferences
- Marque "Receive Billing Alerts"
- Save preferences

**3. Criar Alarm:**
- CloudWatch → Alarms → Create alarm
- Select metric → Billing → Total Estimated Charge
- Currency: USD
- Statistic: Maximum
- Period: 6 hours

**4. Configurar Threshold:**
- Threshold type: Static
- Whenever EstimatedCharges is: Greater than
- **Valor: 1** (US$ 1,00)

**5. Configurar Notificação:**
- Create new SNS topic
- Topic name: `billing-alerts`
- Email: seu-email@exemplo.com
- **Confirme o e-mail** na sua caixa de entrada

### 🧠 Entendendo o Free Tier

**Limites principais:**
- **EC2:** 750 horas/mês de t2.micro ou t3.micro
- **RDS:** 750 horas/mês de db.t2.micro
- **S3:** 5GB de armazenamento
- **Lambda:** 1 milhão de requests/mês

**⚠️ Cuidados:**
- Recursos fora do Free Tier são cobrados imediatamente
- Exceder limites gera cobrança
- Alguns serviços não têm Free Tier (ex: NAT Gateway)



## Passo 3: IAM - Criando o Usuário de Operação

### Por que Não Usar a Conta Root?

- **Princípio do menor privilégio**: Cada usuário deve ter apenas as permissões necessárias
- **Auditoria**: Rastrear ações por usuário específico
- **Rotação**: Facilita troca de credenciais
- **Segurança**: Limita impacto de vazamentos

### Criando Usuário IAM

**1. Acesse IAM:**
- Console AWS → IAM → Users → Add users

**2. Configurar Usuário:**
- User name: `usuario-ads`
- Access type: 
  - ✅ **Programmatic access** (para AWS CLI)
  - ✅ **AWS Management Console access** (para console web)
- Console password: Custom password (crie uma senha forte)
- ✅ **Require password reset** (boa prática)

**3. Definir Permissões:**
- Attach existing policies directly
- Procure e selecione: **AdministratorAccess**

> ⚠️ **Nota Didática:** Estamos usando AdministratorAccess apenas para fins de aprendizado. Em produção, use políticas mais restritivas seguindo o princípio do menor privilégio.

**4. Tags (Opcional):**
- Key: `Environment`, Value: `Learning`
- Key: `Course`, Value: `ImplementacaoSoftware`

**5. Review e Create:**
- Revise as configurações
- Create user

**6. Salvar Credenciais:**
- **Access Key ID**: Anote em local seguro
- **Secret Access Key**: Anote em local seguro
- **🚨 NUNCA COMPARTILHE ESSAS CHAVES**
- **🚨 NUNCA COMMITE NO GITHUB**

### 🔐 Diferença: Senha vs Access Key

| Aspecto | Senha do Console | Access Key |
||||
| **Uso** | Login no console web | APIs e CLI |
| **Formato** | Texto legível | ID + Secret |
| **Rotação** | Manual pelo usuário | Manual pelo admin |
| **Escopo** | Console apenas | Programático apenas |



## Passo 4: AWS CLI - A Nuvem via Terminal

### Instalação no Ubuntu/WSL

```bash
# Atualizar sistema
sudo apt update

# Instalar dependências
sudo apt install curl unzip -y

# Baixar AWS CLI v2
curl "https://awscli.amazonaws.com/awscli-exe-linux-x86_64.zip" -o "awscliv2.zip"

# Extrair
unzip awscliv2.zip

# Instalar
sudo ./aws/install

# Verificar instalação
aws --version
```

**Saída esperada:**
```
aws-cli/2.x.x Python/3.x.x Linux/x.x.x exe/x86_64.ubuntu.xx
```

### Configuração do AWS CLI

```bash
aws configure
```

**Parâmetros solicitados:**

**AWS Access Key ID:** 
- Cole o Access Key ID do usuário `usuario-ads`

**AWS Secret Access Key:**
- Cole o Secret Access Key do usuário `usuario-ads`

**Default region name:**
- **us-east-1** (Norte da Virgínia - mais barato, mais serviços)
- **sa-east-1** (São Paulo - menor latência para Brasil)

**Default output format:**
- **json** (mais estruturado para scripts)
- Alternativas: table, text, yaml

### 🌎 Escolhendo a Região

**us-east-1 (Norte da Virgínia):**
- ✅ Mais barata
- ✅ Todos os serviços disponíveis primeiro
- ✅ Maior Free Tier
- ❌ Maior latência do Brasil

**sa-east-1 (São Paulo):**
- ✅ Menor latência
- ✅ Dados ficam no Brasil (compliance)
- ❌ Mais cara
- ❌ Nem todos os serviços disponíveis

**Recomendação para o curso:** us-east-1 (foco em aprendizado e custos)



## Passo 5: Teste de Conectividade

### Validando a Configuração

```bash
aws sts get-caller-identity
```

**Saída esperada:**
```json
{
    "UserId": "AIDACKCEVSQ6C2EXAMPLE",
    "Account": "123456789012",
    "Arn": "arn:aws:iam::123456789012:user/usuario-ads"
}
```

### 🧠 Interpretando o JSON

**UserId:**
- Identificador único do usuário IAM
- Formato: AIDA + string aleatória

**Account:**
- Número da sua conta AWS (12 dígitos)
- Identifica unicamente sua conta

**Arn (Amazon Resource Name):**
- Identificador completo do recurso
- Formato: `arn:aws:iam::ACCOUNT:user/USERNAME`
- Confirma que você está autenticado como o usuário correto

### Testes Adicionais

```bash
# Listar buckets S3 (deve retornar vazio inicialmente)
aws s3 ls

# Listar instâncias EC2
aws ec2 describe-instances --region us-east-1

# Verificar região configurada
aws configure get region

# Verificar todas as configurações
aws configure list
```



## 🔒 Melhores Práticas de Segurança

### Proteção de Credenciais

**✅ FAÇA:**

## 🆘 Troubleshooting Comum

### "Unable to locate credentials"

**Problema:** AWS CLI não encontra credenciais  
**Solução:** Executar `aws configure` novamente

```bash
aws configure list
# Se mostrar <not set>, reconfigurar
aws configure
```

### "Access Denied"

**Problema:** Usuário sem permissões suficientes  
**Solução:** Verificar políticas IAM

```bash
# Verificar usuário atual
aws sts get-caller-identity

# Verificar se é o usuário correto
```

### "Invalid region"

**Problema:** Região não existe ou mal configurada  
**Solução:** Usar região válida

```bash
# Listar regiões disponíveis
aws ec2 describe-regions --query 'Regions[].RegionName'

# Reconfigurar região
aws configure set region us-east-1
```

### Billing Alarm não funciona

**Problema:** Alarm não dispara  
**Soluções:**
- Verificar se está na região us-east-1
- Confirmar e-mail no SNS
- Aguardar até 24h para primeira coleta
